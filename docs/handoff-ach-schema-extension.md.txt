## How to read this document

This document specifies two things:

1. **The ACH tab** — a new tab added to the existing `Artworks` collection in the bernardbolter.com Payload instance. All fields are ACH-specific. No base archive fields are duplicated here.
2. **The `ImageCaptureTechnologies` collection** — a new dependent collection used by the ACH tab's `imageCaptureType` relation field.

Read `artist-archive-schema-final.md` in full before implementing. The same schema-wide rules apply here without exception: no language-suffix variants, no auto-publish, no hardcoded JSON-LD, Payload v3 only, private fields always access-controlled.

**Layer key** — identical to the archive schema:
- `artist` — entered through Art/Official dialogue; Bernard confirms before record commits
- `agent` — filled computationally or via external lookup; shown for Bernard review
- `private` — role-restricted to artist and admin; never in public API response

**Field type key:**
- `CORE` — foundational display and descriptive data
- `COMPUTED` — machine-generated from other fields
- `RELATIONAL` — connections across works, collections, and external knowledge graphs
- `INTENT` — artist-authored; agent may draft but Bernard must confirm in his own words
- `SYSTEM` — infrastructure for AR, UI behaviour, and future features

**Status key:**
- `NEW` — needs implementing; does not exist yet

All fields in this document are `NEW` — this is a greenfield tab.

## Part 1 — ImageCaptureTechnologies Collection

*Dependent collection. No dependencies — implement before the ACH tab.*

This is the controlled vocabulary for photographic and image-making technologies referenced by MoP source photographs. It replaces the simple `imageCaptureType` select field with a proper linked record — meaning each technology carries its own description, date range, and external URIs, all of which flow into JSON-LD and the future photograph timeline UI without storing duplicate data on the artwork.

### 1.1 Fields

| Field | Type | Layer | Field Type | Definition |
|---|---|---|---|---|
| `name` | Text, localized | artist | CORE | Display name. e.g. *Daguerreotype*, *Wet plate collodion*, *Lithograph*, *Aerial photograph*. |
| `slug` | Text | agent | COMPUTED | Auto-generated from `name`. Unique. Never changes after first publication. |
| `description` | Rich text, localized | artist | CORE | What this technology is, how it works, what distinguishes its visual qualities. Agent drafts; Bernard confirms. |
| `approximatePeriodStart` | Number | artist | CORE | Year this technology came into widespread use. Four-digit integer. |
| `approximatePeriodEnd` | Number | artist | CORE | Year it fell out of common use. Null if still active. |
| `exampleImage` | Upload → R2 | artist | CORE | A representative example image illustrating the technology's visual character. Nullable. |
| `exampleImageCredit` | Text | artist | CORE | Attribution string for the example image. |
| `wikidataUri` | Text (URI) | agent | RELATIONAL | Wikidata entity for this technology. e.g. Daguerreotype → `https://www.wikidata.org/entity/Q178227`. Agent looks up; Bernard confirms. |
| `wikipediaUrl` | Text (URI), localized | agent | RELATIONAL | Wikipedia article. Localized — EN and DE editions link to respective language Wikipedia. Agent looks up; Bernard confirms. |

### 1.2 Slug values (initial vocabulary)

Implement these as the starting records. Bernard adds more as the series grows.

```
daguerreotype
ambrotype
wet-plate-collodion
dry-plate
glass-plate
lithograph
engraving
woodblock-print
early-aerial
satellite
digital-photograph
```

### 1.3 Relationships

- `Artworks` (ACH tab) → `ImageCaptureTechnologies` via `imageCaptureType` relation field (many-to-one)
- `SourcePhotograph` group → `ImageCaptureTechnologies` via `imageCaptureType` relation field (many-to-one)

**Art/Official — Group 1 population notes:**

Agent pre-fills `lat`/`lng` by geocoding the `city` and `country` fields from the base record. Bernard confirms the pin placement is correct for the specific location depicted (the painting may show a specific street or building, not the city centre). `cityPlaceholderColor` is computed silently from the city mapping — no confirmation needed. `tourStopCopy` is drawn out in dialogue: *"If someone was standing in front of this painting on a tour stop, what would you want them to know about it — in your own words?"* Never drafted by the agent.

### Group 2 — Overlay & Colour

Fields that drive the painted-field colour rect gesture and AR button colours.

| Field | Type | Layer | Field Type | Definition |
|---|---|---|---|---|
| `overlayColors` | Array of exactly 3 hex strings | agent | COMPUTED | Three colours extracted from the painted fields in this canvas. Ordered deliberately: used for overlay rects (rotating) and AR buttons (fixed positions). Agent extracts from image analysis and proposes; Bernard confirms or swaps individual colours. |
| `overlayRects` | Array of `{ color: hex, x: %, y: %, w: %, h: % }` | agent | COMPUTED | 1–4 positioned colour rectangles. All values as percentage of image dimensions — resolution-independent. Agent suggests positions based on painted field locations in the image; Bernard confirms or adjusts. Maximum 4 rects. |

**Art/Official — Group 2 population notes:**

On `primaryImage` upload, agent runs image analysis and proposes `overlayColors` (3 values from painted field regions, not from the photographic content) and `overlayRects` (rectangles positioned over the painted field areas). Agent presents these visually for Bernard's review. Bernard can swap individual hex values or drag rect positions. The constraint is that colours must come from the painted fields, not the transferred photograph — agent should weight its extraction toward solid-colour regions. Confirmation is required before these are saved.

### Group 3 — Source Photograph

The historical photograph Bernard used as the transfer source. This group is the richest data section — Art/Official should treat populating it as a research task, not a form-fill.

| Field | Type | Layer | Field Type | Definition |
|---|---|---|---|---|
| `sourceImage` | Upload → R2 | artist | CORE | The source photograph itself — the image Bernard transferred to canvas. Bernard uploads. |
| `sourceImageAltText` | Text, localized | agent | CORE | Alt text for the source photograph. Agent drafts; Bernard confirms. |
| `sourceTitle` | Text, localized | agent | CORE | Title of the photograph as known in the archive or catalogue. Agent looks up from Commons/institution; Bernard confirms. |
| `sourceCreator` | Text | agent | CORE | Photographer name, or institution name if individual creator unknown. |
| `sourceCreatorWikidataUri` | Text (URI) | agent | RELATIONAL | Wikidata entity for the photographer or creating institution. e.g. a named daguerreotypist → their Wikidata Q-number. Agent looks up; Bernard confirms. |
| `approximateDate` | Text | agent | CORE | Human-readable date string for display. e.g. *c. 1861* or *1895–1900*. Agent drafts from known provenance; Bernard confirms. |
| `approximateDateYear` | Number | agent | COMPUTED | Integer year for timeline computation. Use the earliest year in a range. Never displayed directly. Agent sets from `approximateDate`. |
| `imageCaptureType` | Relation → ImageCaptureTechnologies | agent | RELATIONAL | The technology used to make this source photograph. Agent proposes from image analysis and date; Bernard confirms. |
| `sourceWikidataUri` | Text (URI) | agent | RELATIONAL | Wikidata entity for this specific photograph, if one exists. Many Commons images have Wikidata entries. Agent looks up; Bernard confirms or leaves null. |
| `sourceWikimediaCommonsUrl` | Text (URI) | agent | RELATIONAL | Full URL to the Wikimedia Commons file page. e.g. `https://commons.wikimedia.org/wiki/File:Brandenburg_Gate_1900.jpg`. Agent looks up or Bernard provides. |
| `sourceInstitution` | Text | agent | CORE | Archive, museum, or library holding the original. e.g. *Bundesarchiv*, *Library of Congress*. Agent looks up from Commons metadata; Bernard confirms. |
| `sourceInstitutionWikidataUri` | Text (URI) | agent | RELATIONAL | Wikidata entity for the holding institution. e.g. Bundesarchiv → `https://www.wikidata.org/entity/Q815628`. Agent looks up; Bernard confirms. |
| `sourceInstitutionUrl` | Text (URI) | agent | RELATIONAL | Direct URL to the institution's catalogue record for this photograph, if one exists. Agent looks up; Bernard confirms or leaves null. |
| `sourceLicense` | Select | agent | CORE | License under which the source photograph is used. Values: `cc0` · `cc-by` · `cc-by-sa` · `public-domain` · `other`. Agent determines from Commons license tag; Bernard confirms. |
| `sourceLicenseUrl` | Text (URI) | agent | CORE | URL to the specific license. Agent fills from Commons; Bernard confirms. |
| `sourceCredit` | Text | agent | COMPUTED | Full attribution string for display, auto-assembled from the structured fields above. Format: *[Creator], [Title], [Date]. [Institution]. [License].* Agent generates; Bernard may edit for clarity. |

**Art/Official — Group 3 population notes:**

Bernard provides the source image file and, if he has it, the Commons URL. From the Commons URL, agent fetches the file page and extracts: creator, date, institution, license, Wikidata file entity, institution name. Agent then looks up Wikidata entities for creator and institution separately. Agent assembles `sourceCredit` from these structured fields. Agent proposes `imageCaptureType` based on the date and visual character of the photograph — a c.1860 photograph is almost certainly daguerreotype or wet plate; a c.1895 is likely dry plate or glass plate. Bernard confirms. If Bernard doesn't have the Commons URL but knows the photograph, agent can search Commons by description. The goal is that Bernard uploads the image, shares what he knows, and agent does the research — not the other way around.

| `keyHistoricalDates` | Array of `{ year: number, event: localized text }` | artist | INTENT | 3–5 dates Bernard considers anchoring for this location's history as it relates to the work. Not comprehensive history — editorial selection. Agent does not propose these. Drawn out in dialogue. |
| `conceptCopy` | Rich text, localized | artist | INTENT | Bernard's contextual text about this specific artwork. For standalone works: what drove this painting. **Art/Official — Group 4 population notes:**

Agent looks up `locationWikidataUri` from the painting's subject (not just the city — Brandenburg Gate, not Berlin). Agent then looks up `locationTGNUri` from the same subject via the Getty LOD endpoint. Both are confirmed by Bernard. Agent fetches both EN and DE Wikipedia articles and presents them; Bernard confirms the URLs. For `wikipediaExcerpt`, agent presents 4–6 candidate passages from the article — not the introduction, but passages that touch on historical change, visual character, or political significance. Bernard reads and selects. `keyHistoricalDates` and `conceptCopy` are never drafted by agent — drawn out in dialogue: *"What are the dates that matter most to understanding this place in relation to the painting?"* and *"What's the conceptual core of this work — what were you thinking about when you made it?"*

### Group 5 — Reveal Slider

Fields for the three-state reveal slider: source photograph → transfer on canvas → finished painting.

The finished painting is `primaryImage` on the base Artwork record. The slider adds the two intermediate states.

| Field | Type | Layer | Field Type | Definition |
|---|---|---|---|---|
| `transferImage` | Upload → R2 | artist | CORE | The canvas with photo transfer only, before painted fields are added. Must match the crop and framing of `primaryImage` exactly — same camera position, same canvas bounds. Nullable — not every artwork has an in-process photograph. |
| `sliderAxis` | Select | artist | SYSTEM | Direction of the reveal slider. Values: `horizontal` · `vertical`. Default `horizontal`. Bernard chooses per painting based on where the painted fields sit — vertical works better when fields are at top/bottom. Nullable — only set when `transferImage` is present. |

**Art/Official — Group 5 population notes:**

Agent prompts Bernard: *"Do you have a photograph of the canvas after the transfer but before you added the painted fields?"* If yes, Bernard uploads it. Agent reminds Bernard that it must match the crop and framing of the primary image exactly. `sliderAxis` is chosen by Bernard — agent can suggest based on the position of painted fields in the finished painting, but this is always Bernard's decision.

## Part 3 — Triptych Collection: Commerce Fields

*Added to the existing `Triptychs` collection. These fields handle all commercial data for MoP triptych originals and print editions. Commerce data lives on the Triptych record — not on individual Artwork records — because originals and prints are always sold as a set of three, never individually.*

### 3.1 Architecture principle

Payload and Vendure are loosely coupled. Only three things cross the boundary:

1. `vendureProductId` stored in Payload — so ACH knows which Vendure product to call for add-to-cart
2. `printAvailableCount` synced back to Payload via webhook — so pages render remaining count without a live Vendure API call on every page load
3. Payload document IDs passed into Vendure order custom fields — so any order can be resolved back to the archive record permanently

Everything else stays on its own side. Prices, inventory, cart, checkout, order history — all Vendure. Artwork data, edition facts, provenance — all Payload. When Vendure is replaced in the future, the archive is completely untouched.

**Why webhook sync rather than live Vendure call for count display:**
Page render hits Payload once — one fast database read, no external dependency. A live Vendure call on every page load adds latency and a failure point. At MoP edition sizes (15 large, 30 small) the window where Payload's count is stale is seconds, only during an active purchase. The live inventory check happens where it matters — at add-to-cart in Vendure, which will correctly reject if the last item just sold.

### 3.2 Fields to add to Triptychs collection

| Field | Type | Layer | Field Type | Definition |
|---|---|---|---|---|
| `vendureProductId` | Text | artist | SYSTEM | Vendure product ID for the original set. Links Payload triptych record to the Vendure product. Set once when the Vendure product is created. |
| `printSets` | Array | artist | CORE | One entry per print edition size. See sub-fields below. |
| `printSets[].size` | Select | artist | CORE | Values: `large` · `small`. Large = A3, Small = A5. |
| `printSets[].edition` | Number | artist | CORE | Total edition size. 15 for large, 30 for small. Permanent archival record — never changes even after sold out. |
| `printSets[].vendureProductId` | Text | artist | SYSTEM | Vendure product ID for this print edition size. |
| `printSets[].printAvailableCount` | Number | agent | COMPUTED | Remaining prints in this edition. Synced from Vendure via webhook on each sale. Never entered manually. Default equals `edition` total at launch. |
| `printEditionReleaseDate` | Date | artist | CORE | Date the print edition went on sale. Permanent archival record. |
| `signedAndNumbered` | Boolean | artist | CORE | Whether prints are hand-signed and numbered by Bernard. Default `true`. Physical object fact — belongs in the permanent record. |
| `originalsSoldDate` | Date | private | CORE | Date the original set sold. Provenance record. Access-controlled — never in public API. |
| `originalsBuyer` | Text | private | CORE | Buyer name or institution. Provenance record. Access-controlled — never in public API. |

### 3.3 Vendure webhook — implementation note for Cursor

When a print sells in Vendure, Vendure fires a webhook to an ACH API endpoint. That endpoint:

1. Receives the Vendure order payload
2. Identifies which `printSets[]` entry to update via `vendureProductId` match
3. Decrements `printAvailableCount` by the quantity sold
4. Saves the updated Triptych record in Payload

```
POST /api/webhooks/vendure-sale
Body: { vendureProductId, quantitySold }
Action: find Triptych where printSets[].vendureProductId matches
        → decrement printSets[].printAvailableCount by quantitySold
        → save record
```

Endpoint must be authenticated — Vendure webhook secret verified before any Payload write.

### 3.4 What NOT to do — commerce fields

- ✗ Do not store prices in Payload — price lives in Vendure only, changes without touching Payload
- ✗ Do not put commerce fields on individual Artwork records — originals and prints are sold as triptych sets only
- ✗ Do not manually update `printAvailableCount` — webhook only
- ✗ Do not make `originalsSoldDate` or `originalsBuyer` readable by the public API — private fields, access-controlled

## Part 4 — SmallPrints Collection

*New collection for the RIBBA set builder. Cross-series — any square painting from the full archive, not MoP-specific. Post-June feature but schema implemented now.*

### 4.1 Purpose

Bernard has approximately 80 square paintings across all series (New York, Amsterdam, San Francisco, Berlin) that exist as small prints. The RIBBA set builder lets a user select any 5 of these and purchase them as a pack at a fixed price. The collection is a curated subset of the archive — not every artwork, only those Bernard has made available as small prints.

### 4.2 Fields

| Field | Type | Layer | Field Type | Definition |
|---|---|---|---|---|
| `artwork` | Relation → Artworks | artist | RELATIONAL | The source painting from the archive. One-to-one. All title, city, series, year, dimension data comes from this relation — nothing duplicated on SmallPrints. |
| `available` | Boolean | artist | SYSTEM | Whether this print is currently available for set building. Default `true`. Set `false` when out of stock. |
| `vendureProductId` | Text | artist | SYSTEM | Vendure product ID. Note: the RIBBA set is one Vendure product with a fixed price. The 5 selected SmallPrint Payload document IDs are passed as custom fields on the Vendure order line — not as separate products. |

**No `image` field** — the `primaryImage` from the related Artwork record is used directly. Bernard confirmed no crop is needed for the small print format. The white border is applied via a print template at fulfilment, not stored in the database.

**No `triptych` relation** — SmallPrints is cross-series. City, series, and all other context come from the `artwork` relation to the base Artwork record.

### 4.3 Validation constraint

The `artwork` relation must point to an Artwork where `orientation: square` (base archive field). Implement as a Payload validate hook on the `artwork` field — reject if the related artwork is not square. This prevents non-square works from being added to the RIBBA collection accidentally.

### 4.4 Vendure order custom fields — RIBBA set

When a user completes a RIBBA set purchase, the 5 selected SmallPrint Payload document IDs are passed as a custom field on the Vendure order line:

```
Order line custom fields: {
  selectedPrintIds: [
    "payload-doc-id-001",
    "payload-doc-id-047",
    "payload-doc-id-023",
    "payload-doc-id-061",
    "payload-doc-id-008"
  ]
}
```

These IDs are stable Payload document IDs — permanently resolvable back to the Artwork record and the full archive. Bernard can look up any order and identify exactly which 5 paintings were purchased. Order confirmation email can be templated in Vendure to list the 5 painting titles by resolving these IDs.

### 4.5 Art/Official — SmallPrints population notes

Adding a painting to SmallPrints is straightforward — Art/Official creates a new SmallPrints record, sets the `artwork` relation to the correct Artwork record, confirms `orientation: square` on that record, sets `available: true`. No other fields to fill. Bernard reviews and confirms before publish.

## Part 5 — JSON-LD Extensions

*Previously Part 3. Renumbered to accommodate Triptych commerce and SmallPrints sections.*

The ACH tab fields extend the `VisualArtwork` JSON-LD generated by the archive. The following additions apply when ACH fields are present.

All JSON-LD is generated programmatically at render time. Never hardcoded.

```json
{
  "@context": "https://schema.org",
  "@type": "VisualArtwork",

  "locationCreated": {
    "@type": "Place",
    "name": "[city]",
    "sameAs": [
      "[locationWikidataUri]",
      "[locationTGNUri]"
    ]
  },

  "isBasedOn": {
    "@type": "Photograph",
    "name": "[sourceTitle]",
    "dateCreated": "[approximateDate]",
    "creator": {
      "@type": "Person",
      "name": "[sourceCreator]",
      "sameAs": "[sourceCreatorWikidataUri]"
    },
    "contentUrl": "[sourceImage R2 URL]",
    "acquireLicensePage": "[sourceLicenseUrl]",
    "creditText": "[sourceCredit]",
    "sameAs": "[sourceWikidataUri]",
    "isPartOf": {
      "@type": "Archive",
      "name": "[sourceInstitution]",
      "sameAs": "[sourceInstitutionWikidataUri]",
      "url": "[sourceInstitutionUrl]"
    }
  },

  "about": {
    "@type": "Place",
    "name": "[location name]",
    "sameAs": "[locationWikidataUri]"
  }
}
```

**Rules for JSON-LD generation — agents must not violate:**

- `locationCreated.sameAs` must be an array containing both Wikidata and TGN URIs when both are present
- `isBasedOn` must only be output when `sourceImage` is present
- `isBasedOn.creator` must be a typed `Person` or `Organization` object — never a plain name string
- `sourceCreatorWikidataUri` maps to `isBasedOn.creator.sameAs`
- All URIs must be full absolute URLs — never relative paths or Wikidata Q-numbers alone
- `isBasedOn` is omitted entirely if no source photograph data is present

### 6.8 What NOT to do

These constraints apply to every agent working on this implementation.

- ✗ Do not create a separate ACH collection — all fields live as a tab on the existing `Artworks` collection
- ✗ Do not duplicate any base archive fields on the ACH tab — `title`, `year`, `medium`, `dimensions`, `series`, `city`, `country`, `status`, `primaryImage` are all on the base record
- ✗ Do not use language-suffix variants anywhere — `conceptCopyDE`, `wikipediaExcerptEN` etc are all wrong. Use `localized: true`
- ✗ Do not make `overlayColors` or `arButtonColors` more than 3 items — enforce array length validation
- ✗ Do not store overlay rect positions as floats or pixels — percentage strings only, resolution-independent
- ✗ Do not set `arEnabled: true` unless `arMarkerFile` is uploaded — add a Payload validate hook that enforces this
- ✗ Do not hardcode `cityPlaceholderColor` — compute from the `city` field value via the mapping in Group 1
- ✗ Do not output `isBasedOn.creator` as a plain string in JSON-LD
- ✗ Do not add `Group 7` fields to artworks outside MoP/MoW series — admin.condition must gate display
- ✗ Do not touch `artist-archive-schema-final.md` base fields — this is an extension only
- ✗ Do not store prices in Payload — prices live in Vendure only
- ✗ Do not put commerce fields on individual Artwork records — triptych originals and prints are sold as sets only, commerce data belongs on the Triptych record
- ✗ Do not manually update `printAvailableCount` — webhook only, read-only in admin
- ✗ Do not make `originalsSoldDate` or `originalsBuyer` readable by the public API
- ✗ Do not add an `image` field to SmallPrints — use `primaryImage` from the artwork relation
- ✗ Do not allow non-square artworks in SmallPrints — validate hook must enforce `orientation: square`
